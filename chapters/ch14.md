# Ch14 — Scaling Up: GKE Autopilot + Cloud Spanner

## When to Scale Beyond Cloud Run
| Limitation | Solution |
|---|---|
| Need full Kubernetes portability | GKE Autopilot |
| Cloud Run max 100 instances | GKE Autopilot |
| In-memory data exceeds Cloud Run 32GB | Cloud Memorystore (Redis) |
| Database vertical scale limit (Cloud SQL) | Cloud Spanner |
| Requires cloud-agnostic deployment | GKE (standard K8s API) |

## GKE Autopilot

### Key Facts
- Fully managed K8s — no node provisioning, no node pool management
- **Pay per pod** (CPU + memory) not per node (massive dev cost saving)
- Auto-scales pods and underlying nodes
- Built on open-source Kubernetes + Istio service mesh
- Same container images + similar YAML as any K8s platform (EKS, AKS, IKS)

### Create Cluster
```bash
gcloud services enable container.googleapis.com

gcloud container clusters create-auto $CLUSTER_NAME \
  --region=$REGION --project=$PROJECT_ID

# Get credentials
gcloud container clusters get-credentials $CLUSTER_NAME --region=$REGION
```

### Deploy to GKE Autopilot
```yaml
# deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: fact-service
spec:
  replicas: 2
  selector:
    matchLabels:
      app: fact-service
  template:
    metadata:
      labels:
        app: fact-service
    spec:
      serviceAccountName: fact-service-ksa    # Kubernetes SA (for Workload Identity)
      containers:
      - name: fact-service
        image: $REGION-docker.pkg.dev/$PROJECT_ID/$REPO/fact-service:latest
        ports:
        - containerPort: 8080
        resources:
          requests:
            cpu: "250m"
            memory: "512Mi"
```

```bash
kubectl apply -f deployment.yaml
kubectl get pods
kubectl get services
```

## Workload Identity (K8s SA ↔ GCP SA Binding)

Without Workload Identity: mount service account key file (security risk).
With Workload Identity: pods automatically get GCP identity without key files.

```bash
# Enable Workload Identity on cluster (done at creation for Autopilot)

# Create Kubernetes Service Account (KSA)
kubectl create serviceaccount $KSA_NAME --namespace $NAMESPACE

# Create GCP Service Account (GSA) as usual
gcloud iam service-accounts create $GSA_NAME

# Bind KSA → GSA
gcloud iam service-accounts add-iam-policy-binding \
  $GSA_NAME@$PROJECT_ID.iam.gserviceaccount.com \
  --role roles/iam.workloadIdentityUser \
  --member "serviceAccount:$PROJECT_ID.svc.id.goog[$NAMESPACE/$KSA_NAME]"

# Annotate KSA with GSA email
kubectl annotate serviceaccount $KSA_NAME \
  --namespace $NAMESPACE \
  iam.gke.io/gcp-service-account=$GSA_NAME@$PROJECT_ID.iam.gserviceaccount.com
```

## Cloud Spanner

### When to Use
- Horizontal scaling of relational workloads (millions of concurrent users)
- Global distribution with strong consistency
- Write scaling (Cloud SQL only scales reads via replicas)
- Cannot be achieved with single-node RDBMS

### Key Facts
- SQL-compatible (PostgreSQL dialect available)
- Decouples compute from storage (both scale independently)
- Uses TrueTime API (atomic clocks + GPS) for global synchronization
- CAP theorem: prioritises Consistency + Partition tolerance; minimises availability trade-off
- Cost: higher than Cloud SQL but enables global horizontal scaling

### PostgreSQL Compatibility Mode
```bash
gcloud services enable spanner.googleapis.com

# Create Spanner instance (PostgreSQL dialect)
gcloud spanner instances create $INSTANCE_NAME \
  --config=regional-$REGION \
  --processing-units=100 \
  --description="Skills Mapper Spanner"

# Create database (PostgreSQL dialect)
gcloud spanner databases create $DB_NAME \
  --instance=$INSTANCE_NAME \
  --database-dialect=POSTGRESQL
```

### Spanner vs Cloud SQL Decision
| Factor | Cloud SQL | Cloud Spanner |
|---|---|---|
| Scale | Vertical + read replicas | Horizontal (compute + storage) |
| Global consistency | Single region | Multi-region |
| Cost | Lower | Higher |
| Lock-in | Low (PostgreSQL wire) | Higher (Spanner API) |
| Use when | < millions users | Millions of users globally |

### Spanner Compatibility Limitations
Some PostgreSQL features not supported in Spanner; design schema carefully:
- No SERIAL (use UUID or Spanner sequences)
- Interleaved tables for parent-child relationships
- Spanner generates timestamps server-side

## Cloud Memorystore (Redis) for In-Memory Scaling
```bash
gcloud services enable redis.googleapis.com

gcloud redis instances create $INSTANCE_NAME \
  --size=1 --region=$REGION \
  --redis-version=redis_6_x
```

**Use case:** When Cloud Run in-memory data (e.g., trie) grows beyond single instance limits → move to shared Redis cache.

**Principle:** Don't add Memorystore until you actually hit the Cloud Run memory limit — avoid premature complexity.

## Cloud Run Scaling Limits Reference
| Resource | Limit |
|---|---|
| Max instances | 100 per region |
| Max memory per instance | 32 GB |
| Max CPU | 8 vCPU |
| Max concurrency | 1000 per instance |

## Key Scaling Principle
> "Introduce a new technology only when the limitation of the current technology is likely to be reached."

Do not over-engineer from the start. Start with Cloud Run → move to GKE Autopilot
when Kubernetes portability or scale limits are needed.
