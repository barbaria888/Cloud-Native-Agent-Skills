# Ch03 — Cloud Native Applications

## Alan Kay's Vision → Cloud Native
- Software as biological cells: autonomous components communicating by messages
- Cloud native ≈ loosely coupled components communicating via APIs or events
- Innovation: not the idea itself, but the modern toolchain making it practical

## The 12-Factor App (+ 3 Extensions)

Originally for Heroku/Cloud Foundry; adapted for modern platforms.

### Original 12 Factors
| # | Name | Principle | Cloud Native Note |
|---|---|---|---|
| I | Codebase | One repo → many deploys | No env-specific branches |
| II | Dependencies | Explicit + isolated | Containers solve this completely |
| III | Config | In environment | Env vars for non-secret; Secret Manager for secrets |
| IV | Backing services | Attached resources | DB URI in config, not hardcoded |
| V | Build/Release/Run | Strictly separate | Cloud Build (build) → Artifact Registry (release) → Cloud Run (run) |
| VI | Processes | Stateless | No local disk/memory state; store externally |
| VII | Port binding | Self-contained | HTTP on :8080; Cloud Run handles the rest |
| VIII | Concurrency | Scale out | Add instances, not CPU (horizontal > vertical) |
| IX | Disposability | Fast up, graceful down | Handle SIGTERM; finish in-flight requests |
| X | Dev/Prod parity | Keep envs similar | Containers + IaC enforce this |
| XI | Logs | Event streams | Log to stdout; Cloud Logging aggregates |
| XII | Admin processes | One-off tasks | Cloud Scheduler or K8s Jobs |

### 3 Additional Factors (Kevin Hoffman)
| # | Name | Why it matters |
|---|---|---|
| XIII | API First | Clean contracts enable independent service development |
| XIV | Telemetry | Metrics + logs + traces required for cloud-native observability |
| XV | Security | Not an afterthought; least-privilege, scanning, encryption everywhere |

## Microservices

**When to split a monolith:**
1. When change, failure, or scaling need to happen independently
2. When separate teams hit friction sharing code
3. When a piece of functionality hits scaling limits

**Component granularity rule:**
> "If aspects can occur together (change/fail/scale), keep as one unit. When independent behaviour is needed, split."

**Warning:** More fine-grained = more complexity. Automation + observability required to manage it.

## Domain-Driven Design (DDD) for Microservice Boundaries
- **Bounded context**: explicit scope for each microservice's model
- **Domain language**: shared vocabulary within each bounded context
- **Event Storming**: collaborative workshop to map event flows in a domain (sticky notes → bounded contexts)

## Communication Patterns
| Pattern | When to use | GCP Service |
|---|---|---|
| Synchronous REST | External APIs, simple request-reply | Cloud Run + Global LB |
| Synchronous gRPC | Internal services, performance-critical | Cloud Run + gRPC |
| Asynchronous events | Decoupled services, fan-out | Cloud Pub/Sub |
| Hybrid | External REST → internal events | API Gateway + Pub/Sub |

**gRPC vs REST:**
- gRPC: binary (protobuf), rigid contract (.proto), efficient, prefer internally
- REST: JSON, flexible, prefer for external-facing APIs

## Containers
- Packaging + isolation mechanism (both at once)
- OCI standard; Docker is one implementation
- Immutable when locked (no changes at runtime)
- Key for Dev/Prod parity and Dependencies factors
- GCP is container-centric → containers give flexibility across all runtimes

## Container Runtimes (The Car Analogy)
```
Cloud Functions → Taxi (minimal control, just go)
Cloud Run       → Rental car (your drive, no maintenance)
GKE Autopilot   → Uber Black (premium, scalable, no car ownership)
GKE Standard    → Owning a car (full control, full responsibility)
```

## Avoiding Lock-in
- Prefer open standards: PostgreSQL wire protocol, Kubernetes API, Knative, OCI containers
- These services exist in multiple clouds and on-premises

## Development Practices
**Extreme Programming (XP) Principles for Cloud Native:**
- Small releases (maps to microservices increments)
- Test-driven development (TDD) — confidence to change fast
- Refactoring (continuous improvement)
- Collaborative development

**Testing pyramid for cloud native:**
1. Unit tests (isolated function-level)
2. Integration tests (service interactions)
3. **Contract tests** (verify API contracts between services) ← critical for microservices
4. Performance tests (latency under load)
5. Disaster recovery tests
6. Security tests
7. User acceptance tests

**DevOps / Automation eliminating toil:**
- CI: auto build + test on every commit
- CD: auto deploy tested artifacts
- Monitoring: expose inside of Citadel in Observatory
- IaC: reproducible infrastructure
- Managed services: reduce operational burden
