# Ch06 — Project 2: Skill Service with Cloud Run

## Architecture
```
User → GET /autocomplete?prefix=java
     → Cloud Run (skill-service)
     → In-memory Trie (loaded from Cloud Storage tags.csv on startup)
     → JSON response: {"results": ["java", "java-11", ...]}
```

## Why Cloud Run (not Cloud Functions)?
- Service needs to load a ~1MB in-memory trie data structure on startup
- Loading trie per-request would be too slow
- Cloud Run = long-running HTTP service; reuses trie across requests
- Cloud Functions = stateless per-invocation (not appropriate here)

## Cloud Run Key Facts
| Property | Default | Max |
|---|---|---|
| Memory | 512MB | 32GB |
| CPU | 1 vCPU | 8 vCPU |
| Concurrency | 80 | 1000 |
| Min instances | 0 | — |
| Max instances | 100 | — |
| Request timeout | 60s | 60 min |
| Scale-to-zero | Yes | — |

**Cost model:** pay only when processing requests (allocated from Borg on demand).

## Cloud Run Scaling Modes
1. **Multiple instances** — auto-scales based on request queue
2. **Concurrency** — multiple requests per instance (good for Go/Java)
3. **Vertical** — increase CPU/memory per instance

## Deploying from Source
```bash
# Quick deploy (buildpack auto-detects language from go.mod)
gcloud run deploy $SERVICE_NAME --source . \
  --env-vars-file=.env.yaml \
  --allow-unauthenticated

# Get service URL
export URL=$(gcloud run services describe $SERVICE_NAME --format='value(status.url)')

# Test
curl -X GET "$URL/autocomplete?prefix=java"

# Local dev emulator (runs via minikube behind the scenes)
gcloud beta code dev
```

## Buildpacks
- Auto-detect language (go.mod → Go buildpack; pom.xml → Java buildpack)
- No Dockerfile needed
- Cloud Native Buildpacks (CNB) v3 standard
- Two-step shortcut: `gcloud run deploy --source .` = `gcloud builds submit --pack` + `gcloud run deploy --image`

## Service Endpoints (Health Checks)
| Endpoint | Purpose |
|---|---|
| `GET /readiness` | Returns 200 when service can handle requests (liveness) |
| `GET /liveness` | Returns 200 when instance is healthy (readiness) |
| `GET /autocomplete?prefix=X` | Main business endpoint |

Cloud Run monitors readiness/liveness; terminates unhealthy instances.

## Service Account (Least Privilege for Cloud Run)
```bash
gcloud iam service-accounts create $SA_NAME

# Read-only access to Cloud Storage bucket
gsutil iam ch serviceAccount:$SA_NAME@$PROJECT_ID.iam.gserviceaccount.com:objectViewer \
  gs://$BUCKET_NAME

# Allow writing logs to Cloud Logging
gcloud projects add-iam-policy-binding $PROJECT_ID \
  --member=serviceAccount:$SA_NAME@$PROJECT_ID.iam.gserviceaccount.com \
  --role=roles/logging.logWriter

# Apply to service without redeploying
gcloud run services update $SERVICE_NAME \
  --service-account $SA_NAME@$PROJECT_ID.iam.gserviceaccount.com
```

## Performance Tuning (without redeploying)
```bash
# Baseline test: 100 requests from 1 user
ab -n 100 -c 1 -rk "$URL/autocomplete?prefix=java"

# 100 requests from 10 concurrent users (stress test)
ab -n 100 -c 10 -rk "$URL/autocomplete?prefix=java"

# Fix cold start issue: always keep 1 instance warm
gcloud run services update $SERVICE_NAME --min-instances 1

# Increase CPU for CPU-bound work
gcloud run services update $SERVICE_NAME --cpu 2

# Tune concurrency (reduce if CPU is maxed at current concurrency)
gcloud run services update $SERVICE_NAME --concurrency 40
```

**Key metric:** P95 latency (95th percentile) — ensure 95% of requests < 500ms.

## Structured Logging
```bash
# Read logs via Cloud Logging
gcloud beta run services logs read $SERVICE_NAME
gcloud logging read "resource.labels.service_name: $SERVICE_NAME" --limit 1

# Filter for specific log entries
gcloud logging read "resource.labels.service_name: $SERVICE_NAME \
  textPayload: populate tags took" --limit 5 --freshness 1m \
  --format="value(textPayload)"
```

**stdout → INFO, stderr → WARNING** in Cloud Logging automatically.
Structured logging (via client library) adds trace IDs, severity, labels.

## Performance Anti-Patterns
- Not setting min-instances: cold starts add 2+ seconds per scale-up event
- Default concurrency (80) can overload a CPU-bound service
- Not using concurrency for I/O-bound services wastes instance capacity
