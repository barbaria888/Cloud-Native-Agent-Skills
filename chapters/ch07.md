# Ch07 — Project 3: Fact Service with Cloud SQL + Identity Platform

## Architecture
```
Authenticated User → Cloud Run (fact-service: Java/Spring Boot)
                   → Cloud SQL for PostgreSQL (user skill facts)
                   → Secret Manager (DB credentials)
                   → Identity Platform (Firebase Auth) for JWT verification
```

## Fact Service Responsibilities
- CRUD operations for a user's skill facts (add, update, delete skills)
- Authenticates users via Identity Platform JWT tokens
- Persists data in Cloud SQL (PostgreSQL)

## Cloud SQL Key Facts
- Managed PostgreSQL, MySQL, or SQL Server
- Vertical scaling only (increase machine size)
- Can add read replicas (read-only scaling)
- Max ~96 vCPUs, 624 GB RAM per instance
- Supports Cloud SQL Auth Proxy for secure local access

## Cloud SQL Setup
```bash
gcloud services enable sqladmin.googleapis.com

# Create instance (db-f1-micro for dev)
gcloud sql instances create $INSTANCE_NAME \
  --database-version=POSTGRES_14 --tier=db-f1-micro \
  --region=$REGION --project=$PROJECT_ID

# Create database
gcloud sql databases create $DB_NAME --instance=$INSTANCE_NAME

# Create user
gcloud sql users create $DB_USER \
  --instance=$INSTANCE_NAME --password=$DB_PASSWORD

# Connect via Cloud Shell proxy
gcloud sql connect $INSTANCE_NAME --user=$DB_USER
```

## Secret Manager for DB Credentials
```bash
gcloud services enable secretmanager.googleapis.com

# Store secret
echo -n "$DB_PASSWORD" | gcloud secrets create db-password --data-file=-

# Grant access to service
gcloud secrets add-iam-policy-binding db-password \
  --member=serviceAccount:$SA_NAME@$PROJECT_ID.iam.gserviceaccount.com \
  --role=roles/secretmanager.secretAccessor

# Access secret in app (via client library or env var injection)
gcloud secrets versions access latest --secret=db-password
```

## Identity Platform (Firebase Auth)
- Google's managed authentication service
- Supports: email/password, Google, GitHub, SAML, OIDC, phone
- Issues JWT tokens; verified by the backend using public keys
- Cloud Run service validates Bearer token from Authorization header

```bash
gcloud services enable identitytoolkit.googleapis.com

# Enable in Firebase console or via gcloud
# Frontend uses Firebase SDK to sign in
# Backend validates token: verify JWT signature + audience
```

## Spring Boot Integration Patterns
```yaml
# application.yaml (config stored as Cloud SQL instance connection name)
spring:
  datasource:
    url: jdbc:postgresql:///${DB_NAME}?cloudSqlInstance=${INSTANCE_CONNECTION_NAME}&socketFactory=com.google.cloud.sql.postgres.SocketFactory
    username: ${DB_USER}
    password: ${DB_PASSWORD}  # injected from Secret Manager
```

```java
// Spring Cloud GCP auto-configures Cloud SQL connection
// Use @Value to inject secrets from Secret Manager via Spring binding
```

## Service Account Permissions for Fact Service
```bash
# Cloud SQL client
gcloud projects add-iam-policy-binding $PROJECT_ID \
  --member=serviceAccount:$SA@$PROJECT_ID.iam.gserviceaccount.com \
  --role=roles/cloudsql.client

# Secret Manager accessor
gcloud secrets add-iam-policy-binding db-password \
  --member=serviceAccount:$SA@$PROJECT_ID.iam.gserviceaccount.com \
  --role=roles/secretmanager.secretAccessor

# Cloud Logging writer
gcloud projects add-iam-policy-binding $PROJECT_ID \
  --member=serviceAccount:$SA@$PROJECT_ID.iam.gserviceaccount.com \
  --role=roles/logging.logWriter
```

## Deployment
```bash
gcloud run deploy $SERVICE_NAME --source . \
  --env-vars-file=.env.yaml \
  --no-allow-unauthenticated \
  --service-account=$SA@$PROJECT_ID.iam.gserviceaccount.com
```

## Key Design Decisions
- **Stateless service**: all state in Cloud SQL; service can scale horizontally
- **Secrets never in code**: Secret Manager + IAM binding
- **JWT validation in service**: Identity Platform provides public key endpoint
- **Cloud SQL Auth Proxy**: no need to open public IP; uses IAM for auth
