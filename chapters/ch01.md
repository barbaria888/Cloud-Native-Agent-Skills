# Ch01 — What Is Cloud Native?

## Core Distinction
- **Cloud hosted** = WHERE the app runs (on cloud infrastructure)
- **Cloud native** = HOW the app is built (design methodology)
- An app can be cloud-native on-premises (OpenShift) or cloud-hosted-but-not-native (lift-and-shift)

## Why Cloud ≠ Data Center
- Hardware is commodity, generic, designed for failure
- Trust-nothing approach (no implicit perimeter)
- Pay-per-use, elastic, self-service
- Transitioning without adapting → higher cost, higher risk, lower reliability

## 8 Fallacies of Distributed Computing (Deutsch)
1. Network is reliable
2. Latency is zero
3. Bandwidth is infinite
4. Network is secure
5. Topology doesn't change
6. There is one administrator
7. Transport cost is zero
8. Network is homogeneous

Cloud providers build abstractions to address these. Cloud native exploits those abstractions.

## CNCF Definition (Key Part)
> "These techniques enable **loosely coupled systems that are resilient, manageable, and observable**. Combined with robust automation, they allow engineers to make high-impact changes **frequently and predictably** with minimal toil."

The **techniques** (second part of definition) matter more than the technologies (containers, K8s, etc.).

## Cloud Native Goals
- Applications easy to modify
- Apps more reliable than the infrastructure they run on
- Zero-trust security model
- Short feedback cycles + zero downtime

## The Four Facilities Mental Model
```
Cloud
  ├── Laboratory  (developer productivity + experimentation)
  ├── Factory     (automated assembly + testing)
  ├── Citadel     (secure production runtime)
  └── Observatory (system oversight + alerting)
```

All four are needed. Factory-only → kills innovation. Lab-only → no quality.

## Digital Native Advantage
- Unencumbered by legacy infrastructure
- Can start cloud-native from day one
- GCP powers 70% of tech unicorns (10% market share) — resonates with digital natives
