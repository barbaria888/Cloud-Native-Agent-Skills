## Essential gcloud Commands
```bash
# Authenticate
gcloud auth login
gcloud components update       # keep CLI current

# Set defaults
gcloud config set compute/region europe-west2
gcloud config set compute/zone europe-west2-a
gcloud config set run/region europe-west2

# Interactive mode with completion
gcloud beta interactive
```

## Project Management
```bash
export PROJECT_ID=[your-project-id]
gcloud projects create $PROJECT_ID
gcloud config set project $PROJECT_ID

# Enable billing
export BILLING_ACCOUNT_ID=[id-from: gcloud beta billing accounts list]
gcloud beta billing projects link $PROJECT_ID --billing-account $BILLING_ACCOUNT_ID
gcloud beta billing projects describe $PROJECT_ID --format="value(billingEnabled)"
# Unlink billing (stops charges, shuts down services)
gcloud beta billing projects unlink $PROJECT_ID

# Check current project
gcloud config list project
export PROJECT_ID=$(gcloud config get project)
```

## IAM Core Concepts
- **Principal** (user or service) → has **Roles** → that grant **Permissions** on **Resources**
- IAM changes are **eventually consistent** (can take minutes to propagate)
- Every GCP resource needs explicit permission — no implicit trust

## Service Accounts (Key Pattern)
```bash
# Create service account
gcloud iam service-accounts create $SA_NAME \
  --display-name "Description of service account"

# Grant role at project level
gcloud projects add-iam-policy-binding $PROJECT_ID \
  --member=serviceAccount:$SA_NAME@$PROJECT_ID.iam.gserviceaccount.com \
  --role=roles/bigquery.jobUser

# Grant role at resource level (bucket)
gsutil iam ch serviceAccount:$SA_NAME@$PROJECT_ID.iam.gserviceaccount.com:objectAdmin \
  gs://$BUCKET_NAME

# List service accounts
gcloud iam service-accounts list
```

**Best practice:** One dedicated service account per service, with minimum required permissions.

## Environment Variable Pattern
```bash
# Store project config in .env file (NOT in git)
export PROJECT_ID=my-project
export REGION=europe-west2
export BUCKET_NAME=${PROJECT_ID}-skills-data

# Use .env.template committed to git; .env is gitignored
# Load: set -a; source .env; set +a

# Use envsubst to substitute into config templates
envsubst < env.yaml.template > env.yaml
```

## GCP Resource Hierarchy
```
Organization
  └── Folder (optional grouping)
        └── Project  ← billing + IAM boundary
              └── Resources (Cloud Run services, GCS buckets, etc.)
```

## Regional vs Zonal vs Global Resources
- **Zonal**: applies to a single zone (Compute Engine VM, GKE node)
- **Regional**: applies to all zones in a region (Cloud Run, GKE cluster)
- **Global**: available everywhere (Global LB, IAM policies)
