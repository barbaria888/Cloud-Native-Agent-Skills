# Ch13 — Building the Observatory

## The Four Golden Signals (SRE)
| Signal | What it measures | GCP Metric Example |
|---|---|---|
| **Latency** | Time to handle requests (P50/P90/P99) | `run.googleapis.com/request_latencies` |
| **Traffic** | Request rate (throughput) | `run.googleapis.com/request_count` |
| **Errors** | Error rate (4xx/5xx %) | `run.googleapis.com/request_count` filtered by status |
| **Saturation** | Resource utilisation (CPU/memory/instances) | `run.googleapis.com/container/instance_count` |

## Cloud Logging

### Key Commands
```bash
# Read logs with filter
gcloud logging read "resource.labels.service_name: $SERVICE_NAME" --limit 10

# Filter by log content
gcloud logging read \
  "resource.labels.service_name: $SERVICE_NAME AND textPayload: populate tags took" \
  --limit 5 --freshness 1m --format="value(textPayload)"

# Streaming logs
gcloud beta run services logs tail $SERVICE_NAME --project $PROJECT_ID
```

### Log Levels
- `stdout` → INFO in Cloud Logging
- `stderr` → WARNING in Cloud Logging
- Use client library for structured logging (severity levels, labels, trace IDs)

### Log Query Language (LQL)
```
resource.labels.service_name="skill-service"
  AND severity >= WARNING
  AND jsonPayload.message=~"error.*"
```

## Cloud Monitoring

### Built-in Metrics (auto-collected)
- Hundreds of metrics per service (Cloud Run, Cloud SQL, Pub/Sub, etc.)
- Accessible via console → service → Metrics tab or via API

### Custom Dashboards (YAML)
```yaml
displayName: Skill Service Dashboard
gridLayout:
  widgets:
  - text:
      content: Hello World
```

```bash
# Create dashboard
gcloud monitoring dashboards create --config-from-file=dashboard.yaml

# Export existing dashboard to YAML
export DASHBOARD_ID=7fdd430f-4fa1-4f43-aea6-5f8bd9b865df
gcloud alpha monitoring dashboards describe \
  projects/$PROJECT_NUMBER/dashboards/$DASHBOARD_ID > dashboard.yaml
```

### The Four Golden Signals Dashboard (4 widgets)
1. **Container instance count** → saturation
2. **Request latencies** (P50, P90, P99) → latency
3. **Request count** → traffic
4. **Percentage unsuccessful** → errors

## Log-Based Metrics
```bash
# Count metric (how many times a pattern appears in logs)
gcloud logging metrics create tag_suggestion_count \
  --config-from-file=tag_suggestion_count.json

# Distribution metric (extract numeric value from log)
# Uses REGEXP_EXTRACT to parse latency from log message
gcloud logging metrics create tag_suggestion_times \
  --config-from-file=tag_suggestion_times.json

gcloud logging metrics list
```

**Note:** Log-based metrics only collect from the creation time forward (no historical data).

## Alerting
```bash
# Create notification channel (email)
gcloud alpha monitoring channels create \
  --channel-content-from-file=email_notification_channel.yaml

export CHANNEL_ID=$(gcloud alpha monitoring channels list \
  --format="value(name.split('/').slice(-1))")

# Create alert policy
gcloud alpha monitoring policies create \
  --policy-from-file=tags_loaded_number_alert.json
```

**Alert example:** Trigger if tag count < 60,000 (normally ~63,000).

## OpenTelemetry (User-Defined Metrics + Tracing)
```bash
# Grant tracing permission
gcloud projects add-iam-policy-binding $PROJECT_ID \
  --member=serviceAccount:$SA@$PROJECT_ID.iam.gserviceaccount.com \
  --role=roles/cloudtrace.agent
```

**Trace concepts:**
- **Trace**: end-to-end request journey
- **Span**: single timed step within a trace
- `X-Cloud-Trace-Context` header: added to all Cloud Run HTTP requests automatically

**Cloud Run auto-tracing:** All HTTP requests to Cloud Run get a trace ID by default.
Add spans via OpenTelemetry SDK to see internal breakdown.

```go
// OpenTelemetry Go example
tracer := otel.Tracer("skill-service")
ctx, span := tracer.Start(ctx, "autocomplete-search")
defer span.End()
// ... do work
```

## Cost Notes
- First 50 GiB/month of logs: free; after that $0.50/GiB (30-day retention default)
- First 150 MiB/month monitoring data: free per billing account
- Log-based metrics: chargeable beyond free tier
- Traces: first 2.5M spans/month free per project
- Keep retention short unless compliance requires longer periods

## Structured Logging vs Plain Logging
| Plain (stdout) | Structured (client library) |
|---|---|
| Simple text | JSON with fields |
| Severity inferred | Explicit severity |
| No trace correlation | Trace ID + span ID linked |
| Hard to query | Easy to query specific fields |
