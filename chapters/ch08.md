# Ch08 — Project 4: Profile Service with Firestore + Pub/Sub

## Architecture
```
fact-service writes Skill Fact
  → publishes event to Cloud Pub/Sub topic
  → profile-service consumes event (Pub/Sub push subscription)
  → profile-service writes/updates user profile in Cloud Firestore
```

## Services Used
| Service | Role |
|---|---|
| Cloud Pub/Sub | Async event bus between fact-service and profile-service |
| Cloud Firestore | NoSQL document store for user profiles |
| Cloud Run | Runtime for profile-service |
| Secret Manager | Store service credentials |

## Cloud Pub/Sub Key Concepts
- **Topic**: named channel where messages are published
- **Subscription**: named pull/push receiver for a topic
- **At-least-once delivery**: messages may be delivered more than once → make consumers idempotent
- **Push subscription**: Pub/Sub sends HTTP POST to your endpoint
- **Pull subscription**: service polls for messages

```bash
gcloud services enable pubsub.googleapis.com

# Create topic
gcloud pubsub topics create $TOPIC_NAME

# Create push subscription to Cloud Run service
gcloud pubsub subscriptions create $SUB_NAME \
  --topic=$TOPIC_NAME \
  --push-endpoint=$SERVICE_URL/events \
  --ack-deadline=60

# Grant Pub/Sub permission to invoke Cloud Run
gcloud run services add-iam-policy-binding $SERVICE_NAME \
  --member=serviceAccount:service-$PROJECT_NUMBER@gcp-sa-pubsub.iam.gserviceaccount.com \
  --role=roles/run.invoker

# Grant service account publish permission
gcloud pubsub topics add-iam-policy-binding $TOPIC_NAME \
  --member=serviceAccount:$SA@$PROJECT_ID.iam.gserviceaccount.com \
  --role=roles/pubsub.publisher
```

## Cloud Firestore Key Facts
- Serverless NoSQL document database
- Data model: Collections → Documents → Fields (nested maps/arrays)
- Strong consistency for single-document reads
- Eventual consistency for queries across collections
- Auto-scales; pay per read/write/delete operation
- Real-time listeners (good for UIs)

```bash
gcloud services enable firestore.googleapis.com

# Create Firestore in native mode (not Datastore mode)
gcloud firestore databases create --region=$REGION
```

## Firestore Data Model (Profile Service)
```
Collection: profiles/
  Document: {userId}/
    Fields:
      email: string
      displayName: string
      skills: array<SkillFact>
      lastUpdated: timestamp
```

## Event-Driven Design Pattern
```
1. Fact service: user adds skill → save to Cloud SQL → publish event to Pub/Sub
2. Profile service: receive event → update Firestore profile document
3. Result: profile is eventually consistent (eventual consistency acceptable here)
```

**Idempotency:** Profile service must handle duplicate Pub/Sub messages gracefully
(Firestore upsert on document ID is naturally idempotent).

## Service Account Permissions (Profile Service)
```bash
# Pub/Sub subscriber
gcloud projects add-iam-policy-binding $PROJECT_ID \
  --member=serviceAccount:$SA@$PROJECT_ID.iam.gserviceaccount.com \
  --role=roles/pubsub.subscriber

# Firestore user (read/write)
gcloud projects add-iam-policy-binding $PROJECT_ID \
  --member=serviceAccount:$SA@$PROJECT_ID.iam.gserviceaccount.com \
  --role=roles/datastore.user

# Cloud Logging
gcloud projects add-iam-policy-binding $PROJECT_ID \
  --member=serviceAccount:$SA@$PROJECT_ID.iam.gserviceaccount.com \
  --role=roles/logging.logWriter
```

## Pub/Sub Local Emulator
```bash
gcloud beta emulators pubsub start --project=$PROJECT_ID
$(gcloud beta emulators pubsub env-init)  # sets PUBSUB_EMULATOR_HOST
```
Client libraries respect `PUBSUB_EMULATOR_HOST` env var automatically.

## Key Design Decisions
- **Pub/Sub decouples** fact-service from profile-service (neither needs to know about the other)
- **Profile is read-optimised**: Firestore is ideal for user profile reads (fast key lookup)
- **Idempotent consumers**: handle at-least-once delivery from Pub/Sub
- **Push vs Pull**: push is simpler for Cloud Run (no polling needed)
